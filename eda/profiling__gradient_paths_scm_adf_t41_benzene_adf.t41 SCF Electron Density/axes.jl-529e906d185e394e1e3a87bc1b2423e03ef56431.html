1 """
2 ro = IdOffsetRange(r::AbstractUnitRange, offset=0)
3
4 Construct an "identity offset range". Numerically, `collect(ro) == collect(r) .+ offset`,
5 with the additional property that `axes(ro, 1) = axes(r, 1) .+ offset`.
6 When `r` starts at 1, then `ro[i] == i` and even `ro[ro] == ro`,
7 i.e., it's the "identity," which is the origin of the "Id" in `IdOffsetRange`.
8
9 # Examples
10
11 The most common case is shifting a range that starts at 1 (either `1:n` or `Base.OneTo(n)`):
12 ```jldoctest ior
13 julia> using OffsetArrays: IdOffsetRange
14
15 julia> ro = IdOffsetRange(1:3, -2)
16 IdOffsetRange(values=-1:1, indices=-1:1)
17
18 julia> axes(ro, 1)
19 IdOffsetRange(values=-1:1, indices=-1:1)
20
21 julia> ro[-1]
22 -1
23
24 julia> ro[3]
25 ERROR: BoundsError: attempt to access 3-element $(IdOffsetRange{Int, UnitRange{Int}}) with indices -1:1 at index [3]
26 ```
27
28 If the range doesn't start at 1, the values may be different from the indices:
29 ```jldoctest ior
30 julia> ro = IdOffsetRange(11:13, -2)
31 IdOffsetRange(values=9:11, indices=-1:1)
32
33 julia> axes(ro, 1) # 11:13 is indexed by 1:3, and the offset is also applied to the axes
34 IdOffsetRange(values=-1:1, indices=-1:1)
35
36 julia> ro[-1]
37 9
38
39 julia> ro[3]
40 ERROR: BoundsError: attempt to access 3-element $(IdOffsetRange{Int, UnitRange{Int}}) with indices -1:1 at index [3]
41 ```
42
43 # Extended help
44
45 Construction/coercion preserves the (shifted) values of the input range, but may modify
46 the indices if required by the specified types. For example,
47
48 r = OffsetArrays.IdOffsetRange{Int,UnitRange{Int}}(3:4)
49
50 has `r[1] == 3` and `r[2] == 4`, whereas
51
52 r = OffsetArrays.IdOffsetRange{Int,Base.OneTo{Int}}(3:4)
53
54 has `r[3] == 3` and `r[4] == 4`, and `r[1]` would throw a `BoundsError`.
55 In this latter case, a shift in the axes was needed because `Base.OneTo` ranges
56 must start with value 1.
57
58 !!! warning
59
60 In the future, *conversion* will preserve both the values and
61 the indices, throwing an error when this is not achievable. For instance,
62
63 r = convert(OffsetArrays.IdOffsetRange{Int,UnitRange{Int}}, 3:4)
64
65 has `r[1] == 3` and `r[2] == 4` and would satisfy `r == 3:4`, whereas
66
67 ```julia
68 julia> convert(OffsetArrays.IdOffsetRange{Int,Base.OneTo{Int}}, 3:4) # future behavior, not present behavior
69 ERROR: ArgumentError: first element must be 1, got 3
70 ```
71
72 where the error will arise because the result could not have the same axes as the input.
73
74 An important corollary is that `typeof(r1)(r2)` and `oftype(r1, r2)` will behave differently:
75 the first coerces `r2` to be of the type of `r1`, whereas the second converts.
76 Developers are urged to future-proof their code by choosing the behavior appropriate for each usage.
