40 Contains a single callback whose `condition` is a continuous function. The callback is triggered when this function evaluates to 0.
41
42 # Arguments
43 - `condition`: This is a function `condition(u,t,integrator)` for declaring when
44 the callback should be used. A callback is initiated if the condition hits
45 `0` within the time interval. See the [Integrator Interface](@ref integrator) documentation for information about `integrator`.
46 - `affect!`: This is the function `affect!(integrator)` where one is allowed to
47 modify the current state of the integrator. If you do not pass an `affect_neg!`
48 function, it is called when `condition` is found to be `0` (at a root) and
49 the cross is either an upcrossing (from negative to positive) or a downcrossing
50 (from positive to negative). You need to explicitly pass `nothing` as the
51 `affect_neg!` argument if it should only be called at upcrossings, e.g.
52 `ContinuousCallback(condition, affect!, nothing)`. For more information on what can
53 be done, see the [Integrator Interface](@ref integrator) manual page. Modifications to
54 `u` are safe in this function.
55 - `affect_neg!=affect!`: This is the function `affect_neg!(integrator)` where one is allowed to
56 modify the current state of the integrator. This is called when `condition` is
57 found to be `0` (at a root) and the cross is a downcrossing (from positive to
58 negative). For more information on what can
59 be done, see the [Integrator Interface](@ref integrator) manual page. Modifications to
60 `u` are safe in this function.
61 - `rootfind=LeftRootFind`: This is a flag to specify the type of rootfinding to do for finding
62 event location. If this is set to `LeftRootfind`, the solution will be backtracked to the point where
63 `condition==0` and if the solution isn't exact, the left limit of root is used. If set to
64 `RightRootFind`, the solution would be set to the right limit of the root. Otherwise, the systems and
65 the `affect!` will occur at `t+dt`. Note that these enums are not exported, and thus one needs to
66 reference them as `SciMLBase.LeftRootFind`, `SciMLBase.RightRootFind`, or `SciMLBase.NoRootFind`.
67 - `interp_points=10`: The number of interpolated points to check the condition. The
68 condition is found by checking whether any interpolation point / endpoint has
69 a different sign. If `interp_points=0`, then conditions will only be noticed if
70 the sign of `condition` is different at `t` than at `t+dt`. This behavior is not
71 robust when the solution is oscillatory, and thus it's recommended that one use
72 some interpolation points (they're cheap to compute!).
73 `0` within the time interval.
74 - `save_positions=(true,true)`: Boolean tuple for whether to save before and after the `affect!`.
75 This saving will occur just before and after the event, only at event times, and
76 does not depend on options like `saveat`, `save_everystep`, etc. (i.e. if
77 `saveat=[1.0,2.0,3.0]`, this can still add a save point at `2.1` if true).
78 For discontinuous changes like a modification to `u` to be
79 handled correctly (without error), one should set `save_positions=(true,true)`.
80 - `idxs=nothing`: The components which will be interpolated into the condition. Defaults
81 to `nothing` which means `u` will be all components.
82 - `initialize`: This is a function `(c,u,t,integrator)` which can be used to initialize
83 the state of the callback `c`. It should modify the argument `c` and the return is
84 ignored.
85 - `finalize`: This is a function `(c,u,t,integrator)` which can be used to finalize
86 the state of the callback `c`. It can modify the argument `c` and the return is ignored.
87 - `abstol=1e-14` & `reltol=0`: These are used to specify a tolerance from zero for the rootfinder:
88 if the starting condition is less than the tolerance from zero, then no root will be detected.
89 This is to stop repeat events happening immediately after a rootfinding event.
90 - `repeat_nudge = 1//100`: This is used to set the next testing point after a
91 previously found zero. Defaults to 1//100, which means after a callback, the next
92 sign check will take place at t + dt*1//100 instead of at t to avoid repeats.

182 This is also a subtype of `AbstractContinuousCallback`. `CallbackSet` is not feasible when you have many callbacks,
183 as it doesn't scale well. For this reason, we have `VectorContinuousCallback` - it allows you to have a single callback for
184 multiple events.

186 # Arguments
187
188 - `condition`: This is a function `condition(out, u, t, integrator)` which should save the condition value in the array `out`
189 at the right index. Maximum index of `out` should be specified in the `len` property of callback. So, this way you can have
190 a chain of `len` events, which would cause the `i`th event to trigger when `out[i] = 0`.
191 - `affect!`: This is a function `affect!(integrator, event_index)` which lets you modify `integrator` and it tells you about
192 which event occurred using `event_idx` i.e. gives you index `i` for which `out[i]` came out to be zero.
193 - `len`: Number of callbacks chained. This is compulsory to be specified.

315 """
316 $(TYPEDEF)
317
318 Multiple callbacks can be chained together to form a `CallbackSet`. A `CallbackSet`
319 is constructed by passing the constructor `ContinuousCallback`, `DiscreteCallback`,
320 `VectorContinuousCallback` or other `CallbackSet` instances:
321
322 CallbackSet(cb1,cb2,cb3)
323
324 You can pass as many callbacks as you like. When the solvers encounter multiple
325 callbacks, the following rules apply:
326
327 * `ContinuousCallback`s and `VectorContinuousCallback`s are applied before `DiscreteCallback`s. (This is because
328 they often implement event-finding that will backtrack the timestep to smaller
329 than `dt`).
330 * For `ContinuousCallback`s and `VectorContinuousCallback`s, the event times are found by rootfinding and only
331 the first `ContinuousCallback` or `VectorContinuousCallback` affect is applied.
332 * The `DiscreteCallback`s are then applied in order. Note that the ordering only
333 matters for the conditions: if a previous callback modifies `u` in such a way
334 that the next callback no longer evaluates condition to `true`, its `affect`
335 will not be applied.
336
337 """
